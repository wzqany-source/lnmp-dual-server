# ADR-001: 为什么采用 Nginx+Apache 架构而非纯 Nginx

## 背景（Context）

项目需要部署一个支持 PHP 的 Web 服务。常见的方案有两种：
- 纯 Nginx + PHP-FPM
- Nginx 反向代理 + Apache + mod_php

## 决策（Decision）

采用 **Nginx（前端）+ Apache（后端）** 的两层架构。

## 备选方案对比（Alternatives）

| 方案 | 优点 | 缺点 | 适用场景 |
|---|---|---|---|
| **纯 Nginx + PHP-FPM** | 性能更高，内存占用低，静态资源处理快 | PHP 模块支持有限，.htaccess 不支持，迁移旧项目需改配置 | 全新项目、性能敏感 |
| **Nginx + Apache + mod_php** | 兼容性最好，mod_php 稳定，.htaccess 支持完善，Apache 模块生态丰富 | 内存占用稍高，多一层代理开销 | 遗留项目迁移、兼容性优先 |

## 结论（Conclusion）

选择 Nginx+Apache 的原因：

1. **兼容性优先**：项目使用 Apache 的 mod_php，这是传统 LAMP 栈的标准配置，开箱即用，无需额外配置 PHP-FPM
2. **职责分离**：Nginx 处理 SSL 终止和静态资源加速，Apache 专注动态 PHP 请求，两层架构便于独立重启和故障隔离
3. **运维可控**：mod_php 稳定成熟，.htaccess 支持完善，适合学习和兼容性优先的场景
